# Defense-in-depth: атака под web_app_demo (Этап 5)

Цель: показать, что **даже если SQL-инъекция технически проходит**, минимальные привилегии роли приложения блокируют разрушительные действия. Это вторая линия обороны поверх безопасного кода.

## Сценарий

Под `postgres` атака `ORDER BY title; DROP TABLE injection_lab.tasks; --` (Этап 2.3) реально снесла таблицу — спасло только `ROLLBACK`. Повторяем ту же атаку под `web_app_demo`.

`SET ROLE web_app_demo` (LOGIN, SELECT на 2 таблицы + UPDATE(status) на tasks).

## Результаты

| Действие | postgres (Этап 2.3) | web_app_demo (Этап 5) |
|---|---|---|
| SELECT (легитимная часть) | успех | успех |
| DROP TABLE injection_lab.tasks | **успех** (таблица удалена, спас ROLLBACK) | **отказ**: `ERROR: must be owner of table tasks` |
| INSERT в users | (не пробовалось) | **отказ**: `permission denied for table users` |
| TRUNCATE tasks | (не пробовалось) | **отказ**: `permission denied for table tasks` |
| UPDATE title=... | (не пробовалось) | **отказ**: `permission denied for table tasks` |
| UPDATE status=... | (не пробовалось) | **успех** (есть колоночный UPDATE(status)) |

## Вывод

Это работает как двойная защита:

1. **Безопасный код** (параметризация, белые списки) не даст инъекции произойти в первую очередь.
2. **Минимальные привилегии** ограничивают ущерб, если код всё-таки оказался уязвимым. DROP/TRUNCATE/INSERT/UPDATE на не разрешённые колонки заблокированы на стороне БД.

Цитата Лекции 9, раздел 3.1: «Если приложение подключается к PostgreSQL под ролью с избыточными привилегиями, последствия SQL-инъекции резко возрастают. Если же роль имеет только SELECT и ограниченный доступ к конкретным таблицам, ущерб может быть существенно меньше».

Важная деталь: `must be owner of table tasks` — это не «нет права DROP», это «DROP может только владелец объекта». В нашем стенде владелец `postgres`, а `web_app_demo` им не является, и право DROP вообще нельзя выдать обычным GRANT'ом — только сменить владельца через ALTER TABLE OWNER TO. Это и есть архитектурная защита от инъекций уровня DDL.

После аборта транзакции (DROP вызвал ошибку → `current transaction is aborted, commands ignored until end of transaction block`) — `ROLLBACK` чистый. Таблица как была с 4 строками, так и осталась.
