# Метод Peer (Этап 6) — теоретический разбор

## Почему не применяем на железе

Хост — Windows 11, PostgreSQL запущен в Docker-контейнере `pg-auth-test` (образ `postgres:15`).

- `peer` использует Unix domain socket и сверяет имя POSIX-пользователя процесса-клиента с именем роли в БД (через `getpeereid`/`SO_PEERCRED`).
- Внутри контейнера такая аутентификация работает (есть `/var/run/postgresql/.s.PGSQL.5432`, есть POSIX-юзеры `postgres`, `root` и т. д.), но с хоста на Windows подключиться через Unix-сокет нельзя — клиент с хоста ходит только по TCP (`-h 127.0.0.1`), а это уже `host`, а не `local`. Для `host`-подключений метод `peer` неприменим — `pg_hba.conf` не разрешит такой комбинации.
- Сценарий «создать на хосте Windows POSIX-пользователя с нужным именем и зайти через peer» физически невозможен: peer — это Unix-only механизм.
- Внутри контейнера демонстрация была бы тривиальной (там уже работает peer по умолчанию для пользователя postgres через unix-сокет), но образовательной ценности относительно методички не добавляет: ровно то же самое уже видно в стандартном правиле строки 89 `local all all trust` — если бы оно было `peer`, то `docker exec -u postgres pg-auth-test psql` уходил бы под peer-проверкой.

Поэтому в `pg_hba.conf` контейнера правила `peer` не вносим. Ниже — как это выглядело бы и что нужно учесть.

## Как работает peer

1. Клиент подключается через Unix domain socket (`type = local` в `pg_hba.conf`).
2. PostgreSQL получает UID процесса-клиента через системный вызов `getpeereid()` (или эквивалент).
3. UID конвертируется в имя POSIX-пользователя через `getpwuid()`.
4. Имя POSIX-пользователя сравнивается с именем роли PostgreSQL, под которым клиент пытается войти.
5. Совпало — доступ открыт без пароля; не совпало — отказ.

## Когда применять

- Локальные системные сервисы и cron-задачи (бэкапы, миграции) на машине вместе с СУБД.
- Подключение администратора по `sudo -u postgres psql` без хранения пароля в скриптах.
- Отказ от паролей в `local`-подключениях вообще — снижает риск утечек.

## Ограничения

- Только локальные подключения (`local` / Unix socket), не работает для `host` (TCP).
- Только Unix-подобные ОС.
- Имя POSIX-пользователя должно совпадать с именем роли (либо настраивается маппинг через `pg_ident.conf`).
- Если приложение работает от имени системного пользователя (например, `www-data`), то и роль в PG должна называться так же или быть смаппированной.

## Примеры правил `pg_hba.conf`

```text
# Только пользователь ОС postgres может зайти как роль postgres
local   all             postgres                                peer

# Любой POSIX-пользователь — под одноимённой ролью PG
local   all             all                                     peer

# Маппинг через pg_ident.conf: системный www-data -> роль appuser
local   all             appuser                                 peer map=web_app
```

Сопутствующий `pg_ident.conf`:
```text
# MAPNAME       SYSTEM-USERNAME         PG-USERNAME
web_app         www-data                appuser
web_app         deploy                  appuser
```

## Краткое сравнение с другими методами

| метод | где применим | пароль нужен | средство проверки |
|---|---|---|---|
| peer | local (Unix socket) | нет | UID процесса |
| ident | host (TCP), редко | нет | внешний ident-сервер |
| scram-sha-256 | local + host | да | SCRAM challenge-response |
| trust | любой | нет | ничего (всегда успех) |
