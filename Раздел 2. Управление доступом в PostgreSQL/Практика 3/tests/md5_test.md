# Тест метода MD5 (Этап 4)

Конфиг: `conf/pg_hba.conf.md5`
Изменение относительно original: общие `host all all 127.0.0.1/32 trust` и `::1/128 trust` закомментированы, добавлены явные `host all user_md5 ... md5`.

## Результаты

| # | Сценарий | Команда | Ожидаемо | Фактически |
|---|---|---|---|---|
| 1 | user_md5 + правильный пароль | `PGPASSWORD='SecurePass123!' psql -h 127.0.0.1 -U user_md5` | успех | **успех**, `current_user=user_md5`, `client=127.0.0.1` |
| 2 | user_md5 + неверный пароль | `PGPASSWORD='WrongPass!' psql -h 127.0.0.1 -U user_md5` | отказ | **отказ**: `FATAL: password authentication failed for user "user_md5"` |
| 3 | user_md5 без пароля | `PGPASSWORD='' psql -h 127.0.0.1 -U user_md5` | отказ | **отказ**: `fe_sendauth: no password supplied` |

## Сообщения об ошибках (для отчёта)

- Серверная сторона (тест 2): `FATAL: password authentication failed for user "user_md5"` — стандартное сообщение PG, не различает «неверный пароль» и «нет пользователя» (защита от user enumeration).
- Клиентская сторона (тест 3): `fe_sendauth: no password supplied` — сервер запросил пароль, клиент не смог его предоставить (PGPASSWORD пуст, неинтерактивный режим).

## Наблюдения

- md5 работает «как ожидаемо»: проверка пароля идёт, неверный пароль блокируется. Слабость метода — не в логике аутентификации, а в криптографии хеша и протоколе обмена (см. README, ответ на вопрос 2).
- Метод правила (`md5` в pg_hba.conf) совпал с типом хранимого хеша пользователя — иначе была бы ситуация из теста 3 Этапа 3.
