# Обоснование архитектуры ролей

## Уровни иерархии

Система спроектирована трёхуровневой:

1. **Роли-контейнеры (NOLOGIN, 7 шт.)** — атомарные «пакеты» прав на конкретные объекты или операции. Не предназначены для логина, никто под ними не подключается. Используются как кубики, из которых собираются роли пользователей.
2. **Пользовательские роли (LOGIN + INHERIT, 3 шт.)** — `app_user`, `app_manager`, `app_admin`. Принимают пароль, наследуют контейнеры. Это «должности».
3. **Конкретные пользователи (LOGIN, 5 шт.)** — `dev_alice`, `pm_bob`, `dev_charlie`, `admin_diana`, `marketing_eve`. Получают одну из ролей-должностей (или, для `marketing_eve`, напрямую контейнер `app_read_all`).

## Почему контейнеры NOLOGIN

Принцип минимальных привилегий (Лекция 4, раздел 6.1) и лучшая практика 6.2: «Создавайте роли без LOGIN для группировки прав, затем назначайте их пользователям». Если контейнер имеет LOGIN — это потенциальная точка входа, которую нужно защищать паролем и аудитом. NOLOGIN полностью исключает прямое подключение.

## Почему INHERIT, а не NOINHERIT

В нашей модели роли-должности должны автоматически получать все права своих контейнеров — пользователь не должен думать про `SET ROLE`. NOINHERIT уместен, когда нужно сознательно разделить «обычные права» и «повышенные», которые активируются вручную (например, для DBA-режима). У нас такого требования нет, поэтому везде INHERIT (по умолчанию).

## Наследование между контейнерами и ролями

- `app_read_reference` → `app_task_worker`: задачи и комменты бессмысленны без чтения справочников отделов/должностей.
- `app_connect` отдельный: его получает каждая LOGIN-роль (через `app_user`/`app_manager`/`app_admin` или напрямую через `app_read_all`). Логически он «корневой».
- `app_user` → `app_manager` → `app_admin`: классическая лестница «обычный → менеджер → админ», каждый уровень добавляет права, ничего не отнимает.

## Особые правила

- **`app_internal_comments` использует колоночные GRANT'ы** `SELECT (is_internal)` и `UPDATE (is_internal)` на `app.comments` (Лекция 5, GRANT с указанием колонок). Это позволяет менеджеру видеть и менять флаг `is_internal`, в то время как обычный пользователь — нет. На самом деле в нашей реализации `app_manager` всё равно получает полный `SELECT, INSERT, UPDATE, DELETE ON comments` напрямую, поэтому `app_internal_comments` остаётся для документации и для случаев, когда менеджеру нужны только эти колоночные права (например, для специальной отчётной роли).
- **`marketing_eve` не получает `app_user`** — у неё только `app_read_all`. Это сознательный выбор: маркетолог должен видеть данные, но ничего не писать. Если завтра нужно дать ей INSERT — добавим, но по умолчанию принцип минимальных привилегий.
- **`app_read_all` ≠ access_logs**: после `GRANT SELECT ON ALL TABLES` выполняется `REVOKE SELECT ON app.access_logs` — логи аудита остаются исключительно за `app_audit_read` и, через него, за `app_admin`. Это было найдено и поправлено по результатам Этапа 9 (тест B4): без REVOKE marketing_eve неожиданно видел access_logs, что нарушает принцип «аудит-trail только для админа».
- **`CONNECTION LIMIT`** убывает с ростом полномочий: 20 для рядовых пользователей, 50 для менеджеров (их обычно меньше, но они держат сессии в инструментах), 10 для админов. У админа меньше всех — это снижает поверхность атаки (Лекция 4, раздел 5.1 + чек-лист 6.5).
- **`CREATEDB` только у `app_admin`** — нужен для расширения схемы и теста CREATE TABLE. SUPERUSER никому не выдаётся (чек-лист).

## Что НЕ делаем

- Не выдаём SUPERUSER никому (даже admin). Все операции через GRANT на конкретные объекты.
- Не выдаём CREATEROLE — управление ролями остаётся за `postgres`.
- Не используем `WITH ADMIN OPTION` при GRANT — пользователи не должны перевыдавать роли друг другу.
- Не строим RLS-политики (это материал Лекции 6).
