# RLS-решения по таблицам схемы app

| Таблица | RLS | Обоснование |
|---|---|---|
| `app.tasks`        | **Да** | Содержит назначения по пользователям и проектам. Видимость различается роль ↔ роль. |
| `app.comments`     | **Да** | Поле `is_internal` критично: обычный пользователь не должен видеть внутренние заметки. RLS — единственное чистое решение на уровне строк. |
| `app.projects`     | **Да** | Проекты привязаны к отделу и владельцу; пользователь видит только проекты, в которых участвует. |
| `app.users`        | **Да** | Каждый сотрудник видит только свою запись; менеджер — свой отдел. |
| `app.departments`  | Нет    | Справочник, открыт всем читателям через GRANT (контейнер app_read_reference). Фильтрация бессмысленна. |
| `app.positions`    | Нет    | Аналогично — справочник. |
| `app.access_logs`  | Нет    | Доступ только у admin через GRANT (app_full_access после REVOKE у app_read_all в Практике 5). RLS избыточен — таблица в принципе невидима для остальных ролей. |
| `app.task_history` | Нет    | Менеджер видит всё через app_history_full (GRANT). Можно добавить RLS для ограничения по проектам — это **возможное расширение**, в данной лабе не делалось ради чистоты модели. |

Цитата Лекции 6, раздел 1.3: «Полный запрет доступа → используйте REVOKE; защита от администратора → шифрование; скрытие столбцов → column-level GRANT». RLS уместен только для **выборочной фильтрации строк**, а не для «всё или ничего». Поэтому справочники и логи остались под GRANT'ами Практики 5.
